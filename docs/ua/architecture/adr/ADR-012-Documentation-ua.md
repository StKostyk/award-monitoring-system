# ADR-012: Вибір framework документації

**Статус**: Прийнято  
**Дата**: 2025-08-21  
**Автор**: Стефан Костик  
**Зацікавлені сторони**: Архітектор проекту, Команда розробки, Технічні письменники

---

## Контекст

Система моніторингу та відстеження нагород потребує комплексної документації, включаючи документацію API, технічну документацію та архітектурну документацію. Документація повинна бути супроводжуваною, версіонованою та підтримувати множинні формати виводу.

### Передумови
- API-first підхід розробки, що вимагає комплексної документації API
- Технічна документація для системної архітектури та розробки
- Презентація портфоліо, що вимагає професійних форматів документації
- Підтримка як англійського, так і українського контенту

---

## Рішення

**OpenAPI 3.1 + Markdown** було обрано як фреймворк документації для Системи моніторингу та відстеження нагород.

### Обраний підхід
- **Документація API**: Специфікації OpenAPI 3.1 з Swagger UI
- **Технічна документація**: Markdown для структурованого технічного письма
- **Генерація**: Автоматизована генерація документації з коду
- **Версіонування**: Документація як код з Git контролем версій

### Обґрунтування
- **API-First розробка**: OpenAPI дозволяє design-first розробку API
- **Професійний вивід**: Markdown підтримує PDF, HTML та інші формати
- **Автоматизація**: Документація генерується з анотацій коду
- **Контроль версій**: Текстові формати добре працюють з Git
- **Промисловий стандарт**: OpenAPI є стандартом для документації REST API

---

## Наслідки

### Позитивні наслідки
- **Документація API**: Комплексна, інтерактивна документація API
- **Автоматизація**: Зменшені ручні зусилля з кодогенерованою документацією
- **Множинні формати**: Професійний PDF та HTML вивід для портфоліо
- **Контроль версій**: Зміни документації відстежуються разом з кодом

### Негативні наслідки
- **Крива навчання**: Синтаксис Markdown та ланцюжок інструментів документації
- **Налаштування інструментів**: Додаткові кроки збірки для генерації документації
- **Обслуговування**: Потреба підтримувати синхронізацію документації з кодом

---

## Розглянуті альтернативи

### Альтернатива 1: Swagger 2.0
- **Переваги**: Широка підтримка, простіша специфікація
- **Недоліки**: Старіший стандарт, менше функцій ніж OpenAPI 3.1
- **Причина відхилення**: OpenAPI 3.1 надає сучасні функції

### Альтернатива 2: AsciiDoc
- **Переваги**: Розширений синтаксис, широке прийняття, інтеграція GitHub
- **Недоліки**: Складніші опції форматування
- **Причина відхилення**: Markdown легше для професійної документації

## Примітки з реалізації

### Технічні вимоги
- **Версія OpenAPI**: 3.1 для останніх функцій та підтримки JSON Schema
- **Інструменти документації**: Springdoc OpenAPI, plain markdown
- **Формати виводу**: HTML, PDF та інтерактивний Swagger UI
- **Інтеграція збірки**: Maven плагіни для генерації документації

### Структура документації
```
docs/
├── api/
│   ├── openapi.yml              # Специфікація OpenAPI
│   └── api-guide.md             # Посібник використання API
├── technical/
│   ├── architecture.md          # Системна архітектура
│   ├── deployment.md            # Посібник розгортання
│   └── development.md           # Налаштування розробки
└── user/
    ├── user-guide.md            # Документація кінцевого користувача
    └── admin-guide.md           # Адміністративний посібник
```

### Документація API
```java
@RestController
@RequestMapping("/api/v1/awards")
@Tag(name = "Awards", description = "Операції управління нагородами")
public class AwardController {
    
    @Operation(
        summary = "Створити нову нагороду",
        description = "Створює новий запис нагороди в системі"
    )
    @ApiResponse(responseCode = "201", description = "Нагорода успішно створена")
    @PostMapping
    public ResponseEntity<Award> createAward(@RequestBody Award award) {
        // Реалізація
    }
}
```
