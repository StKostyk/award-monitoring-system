# ADR-013: Вибір Frontend Framework

**Статус**: Прийнято  
**Дата**: 2025-08-21  
**Автор**: Стефан Костик  
**Зацікавлені сторони**: Архітектор проекту, Frontend команда, UX команда

---

## Контекст

Система моніторингу та відстеження нагород потребує сучасного, корпоративного frontend framework, здатного обробляти складні користувацькі інтерфейси, інтернаціоналізацію та інтеграцію з backend API. Framework повинен підтримувати TypeScript-first розробку та забезпечувати комплексний інструментарій для середовища одного розробника.

### Передумови
- Потреба в комплексному frontend framework з корпоративними функціями
- Вимога до TypeScript-first підходу розробки
- Складні вимоги UI, включаючи форми, таблиці даних та дашборди
- Підтримка інтернаціоналізації для англійської та української мов
- Інтеграція з Spring Boot backend API

### Припущення
- Контекст одного розробника вимагає відмінний інструментарій та документацію
- Корпоративне прийняття та підтримка спільноти критичні для довгострокової життєздатності
- Експертиза TypeScript цінна для демонстрації портфоліо
- Фреймворк повинен підтримувати як традиційні, так і прогресивні веб-додатки

---

## Рішення

**Angular 20+** було обрано як frontend framework для Системи моніторингу та відстеження нагород.

### Обраний підхід
- **Framework**: Angular 20+ (остання стабільна версія)
- **Мова**: TypeScript як основна мова розробки
- **Архітектура**: Компонентно-орієнтована архітектура з сервісами та модулями
- **Інструмент збірки**: Angular CLI для скаффолдингу проекту та управління збіркою

### Обґрунтування
- **Корпоративне прийняття**: Широко прийнятий в корпоративних середовищах з сильною підтримкою Google
- **TypeScript-First**: Нативна підтримка TypeScript з відмінним інструментарієм та типобезпекою
- **Комплексний фреймворк**: Підхід "батарейки включені"(з англ. повністю готовий) з роутингом, HTTP клієнтом, формами та тестуванням
- **Підтримка спільноти**: Велика спільнота, обширна документація та сторонні бібліотеки
- **Довгострокова підтримка**: LTS версії забезпечують стабільність для корпоративних додатків

## Наслідки

### Позитивні наслідки
- **Продуктивність розробника**: Комплексний CLI інструментарій та підтримка IDE
- **Типобезпека**: TypeScript забезпечує перевірку помилок під час компіляції та кращий рефакторинг
- **Корпоративні функції**: Вбудована ін'єкція залежностей, роутинг та HTTP інтерсептори
- **Підтримка тестування**: Комплексний тестувальний фреймворк з Karma та Jasmine
- **Документація**: Відмінна офіційна документація та навчальні ресурси

### Негативні наслідки
- **Крива навчання**: Крутіша крива навчання порівняно з React для розробників нових до Angular
- **Розмір бандлу(пакета)**: Більший початковий розмір пакета порівняно з легковаговими альтернативами
- **Фреймворк Lock-in**: Сильні переконання та конвенції можуть обмежувати гнучкість

### Нейтральні наслідки
- **Цикл релізів**: Регулярні основні релізи версій вимагають періодичних оновлень
- **Екосистема**: Велика екосистема з перевагами та складністю

## Розглянуті альтернативи

### Альтернатива 1: React
- **Переваги**: Велика екосистема, гнучкий, легковаговий, сильний ринок праці
- **Недоліки**: Потребує додаткових бібліотек для повного рішення, менш категоричний
- **Причина відхилення**: Комплексний підхід Angular краще підходить для корпоративних вимог

### Альтернатива 2: Vue.js
- **Переваги**: М'яка крива навчання, хороша продуктивність, гнучка архітектура
- **Недоліки**: Менше корпоративного прийняття, менш комплексний інструментарій
- **Причина відхилення**: Нижче корпоративне прийняття та TypeScript інтеграція

## Примітки з реалізації

### Технічні вимоги
- **Версія Angular**: 20+ для останніх функцій та покращень продуктивності
- **Node.js**: 18+ для Angular CLI та інструментів збірки
- **TypeScript**: Остання версія, підтримувана Angular
- **Менеджер пакетів**: npm або yarn для управління залежностями

### Структура проекту
```
src/
├── app/
│   ├── core/              # Singleton сервіси, guards
│   ├── shared/            # Спільні компоненти, pipes, directives
│   ├── features/          # Feature модулі
│   │   ├── awards/
│   │   ├── users/
│   │   └── reports/
│   ├── layout/            # Layout компоненти
│   └── assets/            # Статичні ресурси
├── environments/          # Конфігурації середовищ
└── styles/               # Глобальні стилі
```

### Налаштування розробки
```bash
npm install -g @angular/cli@latest
ng new award-monitoring-system --routing --style=scss
ng add @angular/material
ng add @ngrx/store
```

## Відповідність та якість

### Наслідки для безпеки
- **CSRF захист**: Вбудований CSRF захист з HTTP інтерсепторами
- **XSS запобігання**: Angular санітизація та безпека шаблонів
- **Інтеграція автентифікації**: Безшовна інтеграція з JWT токенами

### Вплив на продуктивність
- **Оптимізація бандл(пакету)**: Tree shaking та lazy loading для оптимального розміру пакету
- **Виявлення змін**: OnPush стратегія для оптимізації продуктивності
- **Service Workers**: PWA можливості для офлайн функціональності

### Супроводжуваність
- **Організація коду**: Модульна архітектура з чітким розділенням задач
- **Тестування**: Unit тестування з Jasmine/Karma, E2E тестування з Cypress
- **Лінтинг**: Angular ESLint правила для якості коду

## Метрики успіху

### Ключові показники ефективності
- **Розмір Bundle**: < 500KB gzipped для початкового завантаження
- **Час до інтерактивності**: < 3 секунд на 3G з'єднанні
- **Lighthouse Score**: > 90 для продуктивності та доступності

### Моніторинг та сповіщення
- **Core Web Vitals**: Моніторинг LCP, FID та CLS метрик
- **Відстеження помилок**: Інтеграція з сервісом моніторингу помилок
- **Бюджет продуктивності**: Автоматизовані перевірки збільшення розміру bundle
