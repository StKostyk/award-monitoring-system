## Огляд

Цей документ визначає стандарти технічного писемності для всієї проектної документації для забезпечення послідовності, ясності та підтримуваності.

## Структура документа

### Стандартний заголовок

Кожен документ повинен починатися з:

```markdown
# [Назва документа]
## Система моніторингу та відстеження нагород

> **Етап X**: [Назва етапу]  
> **Версія документа**: X.X  
> **Останнє оновлення**: [Місяць Рік]  
> **Автор**: Стефан Костик  

---

## Зміст (для документів >3 розділів)
```

### Організація розділів

1. **Резюме** - Короткий огляд (2-3 речення)
2. **Основний зміст** - Організовано логічними розділами
3. **Посилання** - Посилання на пов'язані документи
4. **Додатки** - Додаткові матеріали (за потреби)

## Стиль написання

### Принципи

| Принцип | Опис |
|---------|------|
| **Ясність** | Використовуйте просту, точну мову |
| **Лаконічність** | Видаляйте непотрібні слова |
| **Послідовність** | Використовуйте однакові терміни скрізь |
| **Повнота** | Охоплюйте всі сценарії |

### Голос та тон

- Використовуйте **активний стан** (бажано): "Система валідує ввід"
- Уникайте пасивного стану де можливо
- Використовуйте **другу особу** для інструкцій: "Натисніть кнопку"
- Використовуйте **теперішній час** для описів: "API повертає JSON"

### Структура речень

- Тримайте речення до 25 слів
- Одна ідея на речення
- Починайте з найважливішої інформації
- Використовуйте маркований список для 3+ елементів

## Стандарти форматування

### Заголовки

```markdown
# Назва документа (H1) - Один на документ
## Основний розділ (H2)
### Підрозділ (H3)
#### Рівень деталей (H4) - Використовуйте помірковано
```

### Блоки коду

Завжди вказуйте мову для підсвічування синтаксису:

```markdown
    ```java
    public class Example {
        // Java код
    }
    ```
```

### Таблиці

Використовуйте таблиці для структурованих даних:

```markdown
| Колонка 1 | Колонка 2 | Колонка 3 |
|-----------|-----------|-----------|
| Дані 1    | Дані 2    | Дані 3    |
```

### Списки

- Використовуйте **маркований список** для невпорядкованих елементів
- Використовуйте **нумерований список** для послідовних кроків
- Тримайте елементи списку паралельними за структурою

### Виділення

| Формат | Використання |
|--------|--------------|
| **Жирний** | Елементи UI, важливі терміни |
| *Курсив* | Перше використання технічних термінів |
| `код` | Команди, імена файлів, змінні |
| > Цитата | Важливі примітки, попередження |

## Термінологія

### Послідовні терміни

| Використовуйте | Не використовуйте |
|----------------|-------------------|
| користувач | клієнт, замовник |
| натисніть | клацніть, вдарте |
| виберіть | оберіть, підберіть |
| введіть | впишіть, внесіть |
| нагорода | досягнення, відзнака |
| подати | відправити, завантажити |

### Технічні терміни

- Визначайте технічні терміни при першому використанні
- Використовуйте абревіатури тільки після визначення: "Application Programming Interface (API)"
- Підтримуйте глосарій для специфічних термінів проекту

### Написання великими літерами

- **Назви продуктів**: Система моніторингу нагород
- **Функції**: Подання нагород, Черга перегляду
- **Ролі**: Системний адмін, Офіцер GDPR
- **Технології**: Spring Boot, PostgreSQL

## Типи документів

```markdown

**Метод**: GET/POST/PUT/DELETE  
**URL**: `/api/v1/resource`  
**Автентифікація**: Обов'язкова/Опціональна

### Параметри
| Назва | Тип | Обов'язковий | Опис |

### Відповідь
| Код | Опис |

### Приклад
```

### Посібники користувача

- Починайте з мети користувача
- Покрокові нумеровані інструкції
- Включайте скріншоти для складного UI
- Додавайте поради з усунення несправностей

### Runbooks

- Чіткі умови спрацювання
- Покрокові процедури
- Очікувані результати для кожного кроку
- Інструкції відкату
- Контактна інформація

## Іменування файлів

### Конвенції

| Тип | Конвенція | Приклад |
|-----|-----------|---------|
| Англ. документи | UPPER_SNAKE_CASE.md | `USER_GUIDE.md` |
| Укр. документи | filename_ua.md | `USER_GUIDE_ua.md` |
| Приклади коду | lowercase-kebab.ext | `example-service.java` |
| Діаграми | lowercase-kebab.puml | `sequence-login.puml` |

### Розташування

```
docs/
├── api/              # API документація
├── operations/       # Runbooks, усунення несправностей
├── user/             # Документація кінцевого користувача
└── ua/               # Українські переклади
    ├── api/
    ├── operations/
    └── user/
```

## Двомовна підтримка

### Англійська (Основна)

- Всі документи створюються англійською спочатку
- Розташовані в директорії `docs/`

### Українська (Переклад)

- Суфікс `_ua` додається до імені файлу
- Розташовані в дзеркальній структурі `docs/ua/`
- Переклад носієм мови
- Технічні терміни можуть залишатися англійською з поясненням українською

### Інструкції з перекладу

- Зберігайте однакову структуру з англійською
- Зберігайте блоки коду без змін
- Перекладайте скріншоти за можливості
- Використовуйте стандартну українську IT термінологію
