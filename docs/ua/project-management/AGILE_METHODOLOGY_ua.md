## Система моніторингу та відстеження нагород

> **Результат Фази 11**: Управління проєктом та Agile-фреймворк  
> **Версія документа**: 1.0  
> **Останнє оновлення**: Грудень 2025  
> **Автор**: Стефан Костик  
> **Обрана методологія**: Solo Scrum (Scrum, адаптований для одного розробника)

## Короткий огляд

Цей документ визначає agile-методологію для розробки Системи моніторингу та відстеження нагород. Враховуючи контекст одного розробника та портфоліо-проєкт, ми приймаємо **Solo Scrum** — полегшену адаптацію принципів Scrum, оптимізовану для індивідуальної продуктивності при збереженні дисципліни корпоративного рівня.

---

## 1. Аналіз вибору фреймворку

### 1.1 Порівняння фреймворків

| **Фреймворк** | **Розмір команди** | **Складність** | **Придатність для одного розробника** | **Рішення** |
|---------------|-------------------|----------------|---------------------------------------|-------------|
| **Scrum** | 3-9 учасників | Середня | Адаптований з модифікаціями | ✅ Обрано |
| **Kanban** | Будь-який | Низька | Добре для підтримки | Часткове впровадження |
| **SAFe** | 50+ учасників | Висока | Надмірно складний для одного | ❌ Відхилено |
| **XP** | 2-12 учасників | Висока | Парне програмування неможливе | ❌ Відхилено |

### 1.2 Обґрунтування вибору: Solo Scrum

**Чому Scrum (адаптований)?**
- **Обмежені за часом ітерації** забезпечують чіткі віхи для демонстрації портфоліо
- **Планування спринту** забезпечує реалістичне управління обсягом
- **Ретроспективи** дозволяють постійне вдосконалення навіть для одного розробника
- **Уточнення беклогу** підтримує пріоритетизовану чергу завдань
- **Доставка інкременту** регулярно демонструє працюючий продукт

**Адаптації для одного розробника:**
- Щоденні стендапи → Щоденний журнал/огляд завдань (5 хв)
- Планування спринту → Самостійна сесія планування (1 година)
- Огляд спринту → Запис демо та оновлення документації
- Ретроспектива → Письмові роздуми та нотатки щодо покращення процесу

## 2. Фреймворк Solo Scrum

### 2.1 Структура спринту

| **Елемент** | **Тривалість** | **Опис** |
|-------------|----------------|----------|
| **Тривалість спринту** | 2 тижні | Збалансована ітерація для значущих інкрементів |
| **Планування** | 1 година | Вибір з беклогу та розбивка завдань |
| **Щоденний огляд** | 15 хв | Перевірка прогресу, виявлення блокерів |
| **Огляд спринту** | 30 хв | Підготовка демо, документування інкременту |
| **Ретроспектива** | 30 хв | Рефлексія процесу та покращення |

### 2.2 Каденція спринту

```
Тиждень 1                       Тиждень 2
┌─────────────────────────┐    ┌─────────────────────────┐
│ Пн: Планування спринту  │    │ Пн: Розробка            │
│ Вт-Пт: Розробка         │    │ Вт-Чт: Розробка         │
│                         │    │ Пт: Огляд + Ретро       │
└─────────────────────────┘    └─────────────────────────┘
```

### 2.3 Церемонії (адаптовані)

#### Планування спринту (1 година)
1. Огляд результатів попереднього спринту
2. Вибір користувацьких історій з пріоритетизованого беклогу
3. Розбивка історій на технічні завдання
4. Оцінка завдань за допомогою чисел Фібоначчі
5. Прийняття зобов'язання щодо мети спринту

#### Щоденний огляд завдань (15 хв)
- Огляд вчорашнього прогресу
- Планування сьогоднішнього фокусу
- Виявлення блокерів та рішень
- Оновлення дошки завдань

#### Огляд спринту (30 хв)
- Демонстрація завершених функцій (запис відео)
- Оновлення документації
- Підготовка матеріалів для портфоліо
- Створення тегу релізу в Git

#### Ретроспектива спринту (30 хв)
Документування в `RETROSPECTIVES.md`:
- Що пройшло добре?
- Що можна покращити?
- Завдання для наступного спринту

## 3. Артефакти

### 3.1 Продуктовий беклог

**Розташування**: GitHub Issues + `docs/project-management/BACKLOG.md`

**Структура**:
- Епіки (великі функції) → Користувацькі історії → Завдання
- Пріоритезація за методом MoSCoW (Must/Should/Could/Won't)
- Оцінка в story points (Фібоначчі: 1, 2, 3, 5, 8, 13, 21)

### 3.2 Беклог спринту

**Розташування**: GitHub Project Board

**Колонки**:
| Колонка | Опис |
|---------|------|
| **Backlog** | Пріоритезовані елементи для майбутніх спринтів |
| **Sprint** | Зобов'язання поточного спринту |
| **In Progress** | Активна робота (ліміт WIP: 3) |
| **Review** | Код-рев'ю/тестування |
| **Done** | Відповідає Definition of Done |

### 3.3 Інкремент

Кожен спринт доставляє:
- Працюючий, протестований код
- Оновлену документацію
- Примітки до релізу
- Готові для портфоліо артефакти

## 4. Елементи Kanban (гібрид)

### 4.1 Ліміти незавершеної роботи (WIP)

| **Колонка** | **Ліміт WIP** | **Обґрунтування** |
|-------------|---------------|-------------------|
| In Progress | 3 | Фокус та зменшення перемикання контексту |
| Review | 2 | Забезпечення якості перед завершенням |
| Blocked | 1 | Необхідне негайне вирішення |

### 4.2 Метрики потоку

Відстеження щотижня:
- **Lead Time**: Беклог → Done (ціль: <5 днів для історій)
- **Cycle Time**: In Progress → Done (ціль: <3 дні)
- **Throughput**: Завершені історії за спринт

## 5. Інструменти та впровадження

### 5.1 Набір інструментів

| **Призначення** | **Інструмент** | **Використання** |
|-----------------|----------------|------------------|
| **Управління беклогом** | GitHub Issues | Користувацькі історії, баги, завдання |
| **Дошка спринту** | GitHub Projects | Kanban-дошка з автоматизацією |
| **Відстеження часу** | Toggl/Ручне | Розрахунок velocity |
| **Документація** | Markdown-файли | Нотатки спринту, ретроспективи |
| **Контроль версій** | Git + GitHub | Код, гілки, релізи |

### 5.2 Мітки GitHub

```
Пріоритет:   p0-critical, p1-high, p2-medium, p3-low
Тип:         epic, story, task, bug, chore
Статус:      blocked, needs-review, ready
Спринт:      sprint-1, sprint-2, ...
Епік:        epic-auth, epic-awards, epic-workflow, ...
```

### 5.3 Стратегія гілок

```
main (готовий до продакшену)
  └── develop (інтеграція)
        ├── feature/US-001-user-registration
        ├── feature/US-003-award-submission
        └── bugfix/fix-validation-error
```

## 6. Визначення готовності (DoR)

Користувацька історія готова до спринту, коли:

- [ ] Користувацька історія відповідає формату: "Як [роль], я хочу [ціль], щоб [вигода]"
- [ ] Критерії приймання чіткі та тестовані
- [ ] Залежності ідентифіковані та вирішені
- [ ] Історія оцінена (≤13 балів, інакше розбити)
- [ ] Технічний підхід зрозумілий
- [ ] Немає блокерів для негайного початку

## 7. Velocity та потужність

### 7.1 Початкова оцінка velocity

| **Фактор** | **Годин/тиждень** | **Примітки** |
|------------|-------------------|--------------|
| Доступний час | 20-30 | Часткова зайнятість/портфоліо-проєкт |
| Фактор фокусу | 70% | Зустрічі, дослідження, перерви |
| **Ефективні години** | 14-21 | Реальний час кодування |

**Цільовий початковий Velocity**: 15-20 story points за спринт

### 7.2 Відстеження velocity

Після 3 спринтів розрахувати:
```
Середній Velocity = (Sprint1 + Sprint2 + Sprint3) / 3
```

Коригувати зобов'язання спринту на основі фактичного velocity.

## 8. Комунікація та звітність

### 8.1 Звіти спринту

Наприкінці кожного спринту створити:
1. **Підсумок спринту** в `docs/sprints/SPRINT-XX.md`
2. **Примітки до релізу** в `CHANGELOG.md`
3. **Демо-відео** (опціонально, для портфоліо)

### 8.2 Оновлення для стейкхолдерів

| **Стейкхолдер** | **Частота** | **Формат** |
|-----------------|-------------|------------|
| Самоперевірка (ретроспектива) | Кожен спринт | Письмова рефлексія |
| Рецензенти портфоліо | Щомісяця | README + демо |
| Проф. Білоскурський | Щомісяця | Резюме прогресу |

## Підсумок

**Обрана методологія**: Solo Scrum з елементами Kanban

**Ключові практики**:
- 2-тижневі спринти з чіткими церемоніями
- Управління беклогом та дошкою через GitHub
- Ліміти WIP для фокусу
- Безперервна документація та створення портфоліо
- Планування на основі velocity після початкових спринтів
