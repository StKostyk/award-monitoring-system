# Фреймворк Збору Вимог

## 1. Огляд Фреймворку

### 1.1 Мета та Цілі
Цей фреймворк надає комплексну методологію для збору, валідації та документування вимог для Системи Моніторингу та Відстеження Нагород. Підхід забезпечує повне покриття стейкхолдерів, зберігаючи при цьому стандарти документації корпоративного рівня.

### 1.2 Категорії Вимог
| Категорія | Опис | Основні Стейкхолдери | Рівень Документації |
|-----------|-----|---------------------|-------------------|
| **Функціональні Вимоги** | Що система повинна робити | Кінцеві Користувачі, Управління | Детальні користувацькі історії з критеріями прийняття |
| **Нефункціональні Вимоги** | Як система повинна працювати | Технічні Команди, Управління | Кількісні специфікації продуктивності |
| **Бізнес-Правила** | Інституційні політики та обмеження | Виконавчий Рівень, Комплаєнс | Формалізована документація правил |
| **Вимоги Комплаєнсу** | Регулятивні та юридичні мандати | Офіцер GDPR, Команда InfoSec | Матриця відображення комплаєнсу |
| **Вимоги Інтеграції** | Зв'язки зовнішніх систем | Технічні Команди, Системні оператори | Технічні специфікації |
| **Вимоги Безпеки** | Захист та контроль доступу | Команда InfoSec, Офіцер GDPR | Документація контролю безпеки |

### 1.3 Життєвий Цикл Вимог
```
Відкриття → Аналіз → Документування → Валідація → Схвалення → Базовий рівень → Управління Змінами
```

## 2. Планування Воркшопів Відкриття

### 2.1 Типи та Цілі Воркшопів

#### **Воркшоп Виконавчого Бачення**
- **Тривалість:** 2 години
- **Учасники:** Ректор, Декан, Завідувач Кафедри, Команда Проекту
- **Цілі:** Визначити стратегічне бачення, критерії успіху, обмеження
- **Результати:** Заява про бачення, метрики успіху, виконавчі вимоги

#### **Воркшоп Бізнес-Процесів**
- **Тривалість:** 3 години (1 сесія)
- **Учасники:** Секретарі Факультетів, Декан, Завідувач Кафедри
- **Цілі:** Змоделювати поточні процеси, виявити больові точки, визначити майбутній стан
- **Результати:** Карти процесів, користувацькі подорожі, вимоги робочих процесів

#### **Воркшоп Технічних Вимог**
- **Тривалість:** 2 години
- **Учасники:** Команда Розробки, Декан, Технічні стейкхолдери
- **Цілі:** Визначити технічні вимоги, потреби інтеграції, обмеження
- **Результати:** Технічні вимоги, архітектурні обмеження, специфікації інтеграції

#### **Воркшоп Користувацького Досвіду**
- **Тривалість:** 2 години
- **Учасники:** Кінцеві Користувачі, Команда Розробки
- **Цілі:** Визначити потреби користувачів, вимоги інтерфейсу, потреби доступності
- **Результати:** Користувацькі персони, вимоги UI/UX, специфікації доступності

### 2.2 Шаблон Планування Воркшопу

#### **Фаза Перед Воркшопом (1 тиждень до)**
- [ ] **Визначення Цілей:** Чіткі цілі воркшопу та критерії успіху
- [ ] **Вибір Учасників:** Правильні стейкхолдери з повноваженнями прийняття рішень
- [ ] **Підготовка Матеріалів:** Фонові документи, шаблони, інструменти
- [ ] **Налаштування Логістики:** Бронювання кімнати, налаштування технології, кейтеринг
- [ ] **Розповсюдження Попереднього Читання:** Фонові матеріали надіслані за 48 годин до

#### **Шаблон Порядку Денного Воркшопу**
```
1. Відкриття та Знайомство (15 хвилин)
   - Привітання та цілі
   - Знайомство учасників
   - Основні правила та очікування

2. Встановлення Контексту (30 хвилин)
   - Огляд проекту та бачення
   - Аналіз поточного стану
   - Огляд критеріїв успіху

3. Відкриття Вимог (90 хвилин)
   - Фасилітований мозковий штурм
   - Вправи з моделювання процесів
   - Пріоритизація вимог
   - Ідентифікація больових точок

4. Перерва (15 хвилин)

5. Аналіз вимог (1-2 години)
   - Категоризація вимог
   - Обговорення техніко-економічного обґрунтування
   - Відображення залежностей
   - Ідентифікація ризику

6. Валідація та Наступні Кроки (30 хвилин)
   - Огляд та підтвердження вимог
   - Призначення елементів дій
   - Планування наступних кроків
   - Заключні зауваження
```

#### **Фаза після семінару (1 тиждень після)**
- [ ] **Документація:** Вичерпні протоколи засідань і фіксація вимог
- [ ] **Перевірка:** Перегляд учасника та підтвердження зафіксованих вимог
- [ ] **Аналіз:** Категоризація вимог і призначення пріоритетів
- [ ] **Розповсюдження:** Підсумкова документація, роздана всім учасникам
- [ ] **Подальше:** Відстежено елементи дій і розпочато наступні кроки

### 2.3 Керівні принципи сприяння семінару

#### **Обов'язки фасилітатора**
- Зберігати увагу на цілях
- Забезпечити збалансовану участь
- Ефективно керувати часом
- Точно фіксувати всі вхідні дані
- Конструктивно вирішувати конфлікти

#### **Методи сприяння**
- **Мозковий штурм:** Відкрита генерація ідей
- **Affinity Mapping:** Групування пов'язаних понять
- **Голосування в крапку (dot voting):** Пріоритетний рейтинг
- **Відображення процесів:** Візуальна документація робочого процесу
- **Аналіз першопричини:** Глибоке дослідження проблеми

#### **Інструменти та матеріали майстерні**
- Фліпчарт папір і маркери
- Стікери (кілька кольорів)
- Інструменти цифрової співпраці (Miro, Mural)
- Ноутбуки/планшети для документації
- Аудіозаписувальна техніка (за згодою)

## 3. Шаблони та Процедури Інтерв'ю

### 3.1 Категорії Інтерв'ю Стейкхолдерів

#### **Шаблон Виконавчого Інтерв'ю**
**Тривалість:** 45-60 хвилин  
**Формат:** Структуроване з відкритими питаннями

**Відкриваючі Питання (5 хвилин)**
- Яка ваша роль стосовно цього проекту?
- Які ваші основні очікування від системи?
- Як цей проект узгоджується з інституційними стратегічними цілями?

**Стратегічні Вимоги (20 хвилин)**
- Які бізнес-результати ця система повинна забезпечити?
- Які ключові метрики успіху ви використовуватимете для оцінки проекту?
- Які обмеження (бюджет, часові рамки, ресурси) нам слід знати?
- Які ризики вас найбільше турбують?
- Як ви будете вимірювати рентабельність інвестицій для цієї інвестиції?

**Функціональні вимоги (15 хвилин)**
- Які конкретні можливості абсолютно необхідні?
- Які поточні больові точки необхідно усунути?
- Які вимоги до інтеграції ви передбачите?
- Які можливості звітності та аналітики вам потрібні?

**Заключні запитання (5 хвилин)**
- Що ми не обговорювали, що важливо для вашого успіху?
- З ким ще нам говорити, щоб завершити наше розуміння?
- Які ваші очікування щодо постійної участі в проекті?

#### **Шаблон Управлінського Інтерв'ю**
**Тривалість:** 60-90 хвилин  
**Формат:** Напівструктуроване з глибокими зануреннями в процеси

**Роль та Контекст (10 хвилин)**
- Опишіть вашу поточну роль та обов'язки
- Проведіть мене через ваш типовий робочий день, пов'язаний з обробкою нагород
- Які системи та інструменти ви зараз використовуєте?

**Аналіз Поточного Стану (30 хвилин)**
- Опишіть поточний робочий процес обробки нагород крок за кроком
- Що працює добре в поточному процесі?
- Які найбільші больові точки та фрустрації?
- Скільки часу ви витрачаєте на завдання, пов'язані з нагородами, щотижня?
- Які помилки чи проблеми трапляються найчастіше?

**Майбутні вимоги (30 хвилин)**
- Опишіть свій ідеальний робочий процес обробки нагород
- Який функціонал заощадив би вам найбільше часу?
- Яку інформацію потрібно бачити в звітах і інформаційних панелях?
- Як мають бути структуровані робочі процеси затвердження?
- Які сповіщення та попередження були б корисними?

**Потреби в техніці та інтеграції (15 хвилин)**
- Які ще системи потрібно інтегрувати з системою нагород?
- Які дані потрібні для імпорту/експорту?
- Які засоби безпеки та контролю доступу потрібні?
- Яке навчання та підтримка знадобляться вашій команді?

**Завершення (5 хвилин)**
- Що для нас найважливіше зробити правильно?
- Що вас хвилює в впровадженні нової системи?

#### **Шаблон Інтерв'ю Кінцевого Користувача**
**Тривалість:** 45 хвилин  
**Формат:** Орієнтований на користувача з проходженнями завдань

**Походження Користувача (10 хвилин)**
- Розкажіть про вашу роль та як довго ви тут працюєте
- Як часто ви зараз подаєте запити нагород?
- Які пристрої ви зазвичай використовуєте для робочих завдань?

**Поточний Досвід (20 хвилин)**
- Проведіть мене через те, як ви зараз подаєте запит нагороди
- Які документи вам зазвичай потрібно надати?
- Що заплутане або фруструюче в поточному процесі?
- Як ви відстежуєте статус своїх запитів?
- Які запитання у вас часто виникають під час процесу?

**Ідеальний досвід (10 хвилин)**
- Якби ви могли розробити ідеальну систему подання нагород, як би вона виглядала?
- Яку інформацію ви хотіли б бачити про свої нагороди?
- Як би ви хотіли отримувати повідомлення про зміни статусу?
- Що змусило б вас бути впевненими, що ваш запит було оброблено належним чином?

**Технічні вподобання (5 хвилин)**
- Ви віддаєте перевагу мобільним або настільним інтерфейсам?
- Які особливості важливі для доступності?
- Наскільки вам комфортно з новими технологіями?

#### **Шаблон технічного інтерв'ю**
**Тривалість:** 90-120 хвилин  
**Формат:** Технічне глибоке занурення з фокусом на архітектурі

**Технічне середовище (15 хвилин)**
- Опишіть поточну технічну інфраструктуру
- Які системи потрібно було б інтегрувати з системою нагород?
- Які поточні стандарти та вимоги безпеки?
- Яких рамок відповідності ви дотримуєтеся?

**Технічні вимоги (45 хвилин)**
- Які вимоги до продуктивності (час відповіді, пропускна здатність)?
- Які вимоги до доступності та надійності?
- Які вимоги до масштабованості для користувачів і обсягу даних?
- Які засоби контролю безпеки необхідно запровадити?
- Які існують вимоги до резервного копіювання та аварійного відновлення?

**Інтеграція та архітектура (30 хвилин)**
- Які системи аутентифікації слід використовувати?
- Які джерела даних потрібно інтегрувати?
- Які зовнішні служби або API потрібні?
- Які існують вимоги до моніторингу та логування?
- Які обмеження розгортання та інфраструктури застосовуються?

**Відповідність і безпека (20 хвилин)**
- Які конкретні вимоги GDPR застосовуються?
- Яка політика збереження даних та архівування існує?
- Який аудит і звітність про комплаєнс потрібні?
- Які процедури реагування на інциденти безпеки потрібні?

**Технічна підтримка (10 хвилин)**
- Які поточні можливості технічного обслуговування та підтримки потрібні?
- Яких стандартів документації необхідно дотримуватися?
- Яке навчання знадобиться технічному персоналу?

### 3.2 Найкращі Практики Інтерв'ю

#### **Керівництва з Підготовки**
- Дослідити походження учасника та роль
- Переглянути відповідну документацію заздалегідь
- Підготувати конкретні питання на основі типу стейкхолдера
- Встановити чіткі очікування щодо часу та результатів
- Підтвердити логістику (розташування, технології, запис)

#### **Проведення інтерв'ю**
- Почніть із налагодження контакту і налаштування контексту
- Використовуйте відкриті запитання, щоб заохотити детальні відповіді
- Дотримуйтесь уточнюючих запитань
- Уникайте навідних запитань, які упереджують відповіді
- Робіть детальні нотатки, навіть якщо записуєте на камеру/диктофон
- Узагальнити ключові моменти для підтвердження розуміння

#### **Процес після інтерв'ю**
- Документуйте висновки протягом 24 годин
- Перевірте розуміння з учасником, якщо це необхідно
- Категоризуйте вимоги за типом і пріоритетом
- Визначте конфлікти або прогалини для подальших дій
- За потреби заплануйте додаткові сесії

## 4. Дизайн та Впровадження Опитувань

### 4.1 Типи та Цілі Опитувань

#### **Опитування Оцінки Потреб Стейкхолдерів**
**Цільова Аудиторія:** Всі стейкхолдери  
**Час:** Ініціація проекту  
**Тривалість:** 10-15 хвилин  
**Ціль:** Базове розуміння потреб та очікувань

**Зразки Питань:**
1. Наскільки ви задоволені поточним процесом управління нагородами? (шкала 1-5)
2. Які ваші топ-3 больові точки з поточною системою?
3. Які функції надали б вам найбільшу цінність? (рангування)
4. Наскільки критичний мобільний доступ для вашої ролі? (шкала 1-5)
5. Які у вас проблеми щодо впровадження нової системи?

#### **Опитування функціональних вимог**
**Цільова аудиторія:** Кінцеві користувачі та управління  
**Таймінг:** Фаза збору вимог  
**Тривалість:** 20-30 хвилин  
**Мета:** Детальне визначення пріоритетів функціональних вимог

**Зразки запитань:**
1. Оцініть важливість кожної функції (критична/важлива/було б добре мати):
   - Автоматизований парсинг документів
   - Сповіщення в реальному часі
   - Розширені можливості пошуку
   - Мобільне представлення
   - Інформаційна панель аналітики
   
2. Для кожного кроку робочого процесу вкажіть свої переваги:
   - Схвалення одним клацанням миші чи детального перегляду
   - Автоматичні сповіщення чи ручна реєстрація
   - Масові операції чи індивідуальні обробки

#### **Опитування технічних вимог**
**Цільова аудиторія:** Технічні зацікавлені сторони  
**Timing:** Фаза планування архітектури  
**Тривалість:** 15-20 хвилин  
**Мета:** Технічне обмеження та ідентифікація вимог

**Зразки запитань:**
1. Оцініть критичність кожної нефункціональної вимоги (шкала 1-5):
   - Час відгуку <200мс
   - 99, 9% безвідмовної роботи
   - Підтримка 1000+ одночасних користувачів
   - Синхронізація даних у реальному часі
   
2. Виберіть відповідні вимоги комплаєнсу:
   - Захист даних GDPR
   - Сліди фінансового аудиту
   - Зберігання академічної документації
   - Звітування про інциденти безпеки

### 4.2 Фреймворк Впровадження Опитувань

#### **Принципи Дизайну Опитувань**
- **Чіткі Цілі:** Кожне питання служить конкретній меті збору вимог
- **Відповідна Довжина:** Баланс між комплексністю та рівнем відповідей
- **Логічний Потік:** Групування пов'язаних питань та використання прогресивного розкриття
- **Варіанти Відповідей:** Надання значущих шкал та варіантів
- **Доступність:** Забезпечення сумісності з допоміжними технологіями

#### **Стратегія Розповсюдження Опитувань**
| Група Стейкхолдерів | Метод Розповсюдження | Цільовий Рівень Відповідей | Розклад Нагадувань |
|---------------------|---------------------|---------------------------|--------------------|
| **Виконавчий** | Персональне запрошення по email | 100% | День 3, День 7 |
| **Управлінський** | Email з підтвердженням керівництва | 90% | День 2, День 5, День 7 |
| **Кінцеві Користувачі** | Кілька каналів (email, оголошення) | 80% | День 3, День 6, День 9 |
| **Технічний** | Пряма комунікація технічної команди | 100% | День 2, День 5 |

#### **Збір та Аналіз Даних**
- **Період Збору:** 10 робочих днів з розкладом нагадувань
- **Моніторинг Відповідей:** Щоденне відстеження з цільовим наслідуванням
- **Якість Даних:** Перевірки достовірності для неповних відповідей
- **Підхід до Аналізу:** Кількісний аналіз з якісним кодуванням
- **Звітність:** Резюме результатів, специфічних для стейкхолдерів

## 6. Валідація та Схвалення Вимог

### 6.1 Методи Валідації

#### **Цикли Огляду Стейкхолдерів**
- **Чернетковий Огляд:** Початковий зворотний зв'язок стейкхолдерів щодо документованих вимог
- **Формальний Огляд:** Структурований огляд з процесом підпису
- **Міжфункціональна Валідація:** Огляд між групами стейкхолдерів на предмет конфліктів
- **Технічна Реалізованість:** Валідація команди архітектури та розробки

#### **Техніки Валідації**
- **Сесії Прогляду:** Покроковий огляд вимог зі стейкхолдерами
- **Валідація Прототипу:** Макети та прототипи для вимог користувацького інтерфейсу
- **Тестування Сценаріїв:** Прогляд випадків використання для вимог робочих процесів
- **Огляд Відповідності:** Юридична та GDPR валідація для вимог відповідності

### 6.2 Робочий Процес Схвалення

```
Документування Вимог → Огляд Стейкхолдерів → Вирішення Конфліктів → 
Технічна Валідація → Схвалення Управління → Виконавчий Підпис → Базова Рівень
```

#### **Критерії Схвалення**
- **Повнота:** Всі ідентифіковані вимоги документовані
- **Ясність:** Вимоги є однозначними та тестованими
- **Узгодженість:** Немає конфліктів між вимогами
- **Реалізованість:** Підтверджена технічна та бізнес-реалізованість
- **Відстежуваність:** Джерело та обґрунтування чітко документовані
